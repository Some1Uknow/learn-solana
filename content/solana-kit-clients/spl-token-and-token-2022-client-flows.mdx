Token flows go wrong when the client guesses.

The app guesses the token program.

It guesses the decimals.

It assumes any token account pair is valid for the chosen mint.

That is how users end up signing transfers that were malformed before the wallet popup even opened.

The correct model is stricter.

Before you ask the wallet to sign, the client should already know:

- which token program you are using
- which mint the transfer is for
- which token accounts belong to that mint
- which raw amount should be transferred

That is the whole lesson.

## Classic SPL Token and Token-2022 Are Different Programs

They are not just two names for the same thing.

They are different program IDs with different extension capabilities.

So the client must keep the token program choice explicit.

Do not hide it inside a helper that silently assumes classic SPL Token.

## Step 1: Build A Transfer Plan With Explicit Program Choice

Install the token clients if you have not already:

```bash
pnpm add @solana-program/token @solana-program/token-2022
```

Now create `lib/solana/tokenFlow.ts`:

```ts
import { address } from "@solana/kit";
import {
  getTransferCheckedInstruction as getSplTransferCheckedInstruction,
} from "@solana-program/token";
import {
  getTransferCheckedInstruction as getToken2022TransferCheckedInstruction,
} from "@solana-program/token-2022";

export type TokenProgramKind = "spl" | "token2022";

export type TokenTransferInput = {
  programKind: TokenProgramKind;
  mint: string;
  sourceTokenAccount: string;
  destinationTokenAccount: string;
  authority: string;
  amountRaw: bigint;
  decimals: number;
};

export function validateTokenTransferInput(input: TokenTransferInput) {
  if (!input.mint) throw new Error("Missing mint address");
  if (!input.sourceTokenAccount) throw new Error("Missing source token account");
  if (!input.destinationTokenAccount) throw new Error("Missing destination token account");
  if (!input.authority) throw new Error("Missing transfer authority");
  if (input.amountRaw <= 0n) throw new Error("Amount must be positive");
  if (input.decimals < 0 || input.decimals > 18) throw new Error("Invalid decimals");
}

export function buildTokenTransferInstruction(input: TokenTransferInput) {
  validateTokenTransferInput(input);

  const mint = address(input.mint);
  const sourceTokenAccount = address(input.sourceTokenAccount);
  const destinationTokenAccount = address(input.destinationTokenAccount);
  const authority = address(input.authority);

  if (input.programKind === "spl") {
    return getSplTransferCheckedInstruction({
      source: sourceTokenAccount,
      mint,
      destination: destinationTokenAccount,
      authority,
      amount: input.amountRaw,
      decimals: input.decimals,
    });
  }

  return getToken2022TransferCheckedInstruction({
    source: sourceTokenAccount,
    mint,
    destination: destinationTokenAccount,
    authority,
    amount: input.amountRaw,
    decimals: input.decimals,
  });
}
```

This file does the two jobs that matter most:

- it rejects obviously broken transfer inputs early
- it makes program choice explicit before instruction assembly

That is what a safe token client should do.

## Why `transferChecked` Matters

`transferChecked` is the right beginner default because the transfer instruction includes decimals explicitly.

That forces the client to be honest about the mint precision it thinks it is dealing with.

If your UI math and mint metadata disagree, the client should fail before that turns into a silent bad transfer.

## Step 2: Preview The Transfer Before Sending

Now create `components/solana/TokenTransferPreview.tsx`:

```tsx
"use client";

import React, { useState } from "react";
import { buildTokenTransferInstruction, type TokenProgramKind } from "@/lib/solana/tokenFlow";

export function TokenTransferPreview() {
  const [programKind, setProgramKind] = useState<TokenProgramKind>("spl");
  const [result, setResult] = useState<string>("");
  const [error, setError] = useState("");

  const onPreview = async () => {
    setError("");

    try {
      const instruction = buildTokenTransferInstruction({
        programKind,
        mint: "ENTER_REAL_MINT_ADDRESS",
        sourceTokenAccount: "ENTER_REAL_SOURCE_TOKEN_ACCOUNT",
        destinationTokenAccount: "ENTER_REAL_DESTINATION_TOKEN_ACCOUNT",
        authority: "ENTER_REAL_AUTHORITY_ADDRESS",
        amountRaw: 1_000_000n,
        decimals: 6,
      });

      setResult(JSON.stringify(instruction, null, 2));
    } catch (e: any) {
      setResult("");
      setError(String(e?.message ?? e));
    }
  };

  return (
    <div className="space-y-3">
      <select
        value={programKind}
        onChange={(e) => setProgramKind(e.target.value as TokenProgramKind)}
        className="px-3 py-2 rounded bg-zinc-900"
      >
        <option value="spl">Classic SPL Token</option>
        <option value="token2022">Token-2022</option>
      </select>

      <button
        onClick={onPreview}
        className="px-3 py-2 rounded bg-indigo-600 text-white"
      >
        Preview Transfer Instruction
      </button>

      {error ? <div className="text-xs text-red-300">{error}</div> : null}
      {result ? <pre className="text-xs bg-zinc-950 p-3 rounded">{result}</pre> : null}
    </div>
  );
}
```

This component still does not send the transfer.

That is correct.

First learn to build the instruction honestly.

Then attach wallet signing and transport.

## The Failure Modes To Avoid

### Assuming classic SPL Token by default

That is how Token-2022 support quietly breaks.

Make the token program part of your input model.

### Treating decimals as a display-only concern

Decimals affect raw amount construction.

They are part of transfer correctness.

### Skipping mint and token-account validation

If the mint and token accounts do not line up, the transfer is already wrong before the wallet signs anything.

## What You Can Do After This

You can build token transfer instructions with explicit program choice, validate the critical inputs early, and use checked transfer builders instead of hand-waving token precision.

That is the client-side baseline for both classic SPL Token and Token-2022 flows.
